# Store Management Homepage Design

## Goal

将 `店铺管理` 从当前的占位页升级为一个真实可用的首页工作台，并保持与现有 `商品列表`、`产品库存`、`批次库存`、`销售管理` 页面一致的中后台工作区风格。

首页不承担完整新增表单，而是承担以下三个核心任务：

- 让用户快速了解全部店铺的运行健康度
- 让用户快速发现并处理授权、同步、配置异常
- 让用户能够顺手进入新增店铺、平台配置和后续详情页面

## Scope

In scope:

- 将 `/stores/list` 从 `PlaceholderView` 替换为真实首页工作台
- 保留左侧导航中的菜单项 `店铺列表`，但页面主标题使用 `店铺管理`
- 复用当前项目已形成的页面结构：
  - 页头说明
  - 指标摘要条
  - 筛选与快捷操作区
  - 主工作区
  - 可配置表格
- 首页支持以下 V1 交互：
  - 点击指标联动表格筛选
  - 点击待处理事项联动表格筛选
  - 从快捷入口进入后续动作
  - 从表格进入详情、重新授权和立即同步
- 使用 mock 数据构建完整页面状态与交互
- 为首页数据处理和关键交互补充测试

Out of scope:

- 接入真实店铺接口
- 在本轮实现 `新增店铺`、`店铺详情`、`平台配置` 的真实业务页面或完整表单
- 实现平台授权向导、API 表单或多步骤接入流程
- 持久化首页筛选条件、列配置或用户偏好
- 改造 `店铺库存`、`利润核算`、`广告推广` 等其他店铺管理二级页面

## Information Architecture

店铺管理模块在 V1 中继续沿用现有左侧导航结构，不新增独立的模块入口。

- 左侧导航项仍显示为 `店铺列表`
- 路由路径保持 `/stores/list`
- 页面主标题改为 `店铺管理`
- 页面定位为 `店铺运营工作台`，而不是普通列表页

首页从上到下分为 4 层：

1. 页头
2. 指标摘要条
3. 待处理事项 + 快捷入口主工作区
4. 店铺列表

这种结构优先回答三个问题：

- 当前整体健康度如何
- 哪些店铺需要立刻处理
- 下一步应进入哪个动作

## Route And View Boundaries

新增一个 route-level view：

- `src/views/StoreManagementHomeView.vue`

该 view 只负责承载首页 workbench 组件，不持有业务状态。

`src/router/index.ts` 需要把 `/stores/list` 从 `PlaceholderView` 改成真实 view 映射，并为该页面设置：

- `sectionKey: 'stores'`
- `sectionTitle: '店铺管理'`
- `hideBreadcrumb: true`

`src/data/navigation.ts` 不调整侧边栏信息架构，继续保留 `店铺列表` 作为当前入口。

## Component Boundaries

### `src/components/stores/StoreManagementWorkbench.vue`

首页主工作台组件，负责：

- 渲染页头、指标摘要条、待处理区、快捷入口和表格
- 持有筛选条件、分页、抽屉开关、当前选中记录、列设置状态
- 处理指标点击、事项点击、表格筛选和快捷操作

### `src/components/stores/StoreHealthActionPanel.vue`

负责首页主工作区上半部分，内部采用左右双栏：

- 左侧为待处理事项列表
- 右侧为快捷入口工具面板

该组件只负责展示与事件抛出，不持有全局数据源。

### `src/components/stores/StoreDetailDrawer.vue`

右侧抽屉，展示单个店铺的轻量详情，用于避免首页表格频繁跳页。

V1 内容包括：

- 基础信息
- 授权与同步状态
- 最近运行摘要
- 快捷动作

### `src/components/stores/storeManagement.css`

首页局部样式文件，延续当前项目 workbench 组织方式：

- 复用现有页面头部、状态 pill、表格容器的视觉语言
- 只增加店铺首页特有的布局和状态样式

不抽象新的全局父类组件，避免为了复用而增加不必要的复杂度。

## Shared UI Rules

店铺管理首页需要严格复用当前工作台设计语言：

- 页头与说明文案采用 `销售管理`、`批次库存` 同样的紧凑结构
- 指标区沿用横向 summary strip，而不是大卡片或营销式面板
- 主工作区卡片圆角、边框、背景层级与现有表格容器一致
- 状态统一使用现有 `status-pill` 语义色，不另起一套颜色系统
- 按钮密度、图标尺寸、间距节奏保持与现有页面一致
- 表格继续使用 `ConfigurableDataTable`

首页不使用以下表现方式：

- 巨型统计卡片墙
- 营销感很强的 hero 区
- 多 tab 切换作为首页主要交互
- 把新增店铺表单直接塞进首页

## Data Layer

新增数据文件：

- `src/data/storeManagement.ts`

负责：

- 定义店铺首页相关类型
- 提供 mock 店铺数据
- 提供摘要统计函数
- 提供异常事项汇总函数
- 提供筛选、排序和状态格式化函数

建议包含以下类型：

- `StoreRecord`
- `StoreHealthMetric`
- `StoreIssueBucket`
- `StoreFilterState`
- `StoreStatus`
- `StoreSyncStatus`
- `StoreAuthorizationStatus`

默认排序规则：

1. 异常店铺优先
2. 待处理授权问题优先
3. 最近活跃或最近同步时间靠前

## Page-Level Design

### Header

- 页面标题：`店铺管理`
- 说明文案：强调统一管理多平台店铺接入、授权状态与同步运行情况
- 右侧主按钮：`新增店铺`
- 次级动作：`批量同步`
- 更多操作收纳为轻量菜单，不堆叠过多按钮

页面头部的作用是明确“这是工作台首页”，而不是某个单一列表页。

### Summary Strip

顶部摘要条放 4 个核心指标：

- 已接入店铺
- 正常运行
- 授权异常
- 同步异常

每个指标都必须可点击，点击后直接联动下面的列表筛选。

指标文案需要带有动作语义，例如：

- `授权异常 3 / 需重新授权`
- `同步异常 5 / 待处理`

### Health And Actions Area

首页第二层是最重要的工作区，使用左右双栏布局。

左侧为 `待处理事项`：

- 授权失效
- API 配置缺失
- 最近同步失败
- 超时未同步

每条事项展示：

- 标题
- 数量
- 简短说明
- 最近更新时间或影响范围

点击事项后，列表应自动进入对应筛选视图。

右侧为 `快捷入口`：

- 新增店铺
- 平台配置
- 同步策略
- 接入说明

快捷入口风格保持克制，更像工具面板，不做大面积插画卡片。

### Filter Bar

店铺列表上方保留轻量筛选，不抢首页的工作台主层级。

V1 筛选项：

- 平台
- 授权状态
- 同步状态
- 站点 / 区域
- 关键词

关键词支持：

- 店铺名称
- 店铺编号
- 平台店铺 ID

筛选区保留：

- 查询
- 重置
- 刷新
- 定制列

布局和交互复用现有库存/商品页的 filter shell。

### Store Table

首页下半部分是完整表格，而不是简单预览。

V1 建议列：

- 平台
- 店铺名称
- 店铺编号
- 授权状态
- 同步状态
- 站点 / 区域
- 最近同步时间
- 近 7 天订单摘要
- 操作

其中：

- `平台` 可用图标 + 文本增强识别
- `店铺名称` 单元格可带副信息，如平台店铺 ID 或负责人
- `近 7 天订单摘要` 使用紧凑双行信息，而不是做复杂图表

操作优先级：

- 详情
- 重新授权
- 立即同步
- 更多

### Detail Drawer

点击表格主操作或店铺名称时，打开右侧抽屉。

V1 抽屉分为三个区块：

- 基础信息：平台、店铺名称、站点、主体、负责人
- 状态信息：授权状态、同步状态、最后同步时间、异常摘要
- 快捷动作：重新授权、立即同步、进入后续配置页

抽屉只做轻量详情，不在本轮塞入复杂 tab。

## Interaction Rules

首页交互遵循“先定位问题，再进入动作”的原则。

- 点击指标时，列表直接切换到对应筛选结果
- 点击待处理事项时，列表直接聚焦相关异常店铺
- 多个联动筛选同时存在时，页面需要明确展示当前筛选条件
- `新增店铺` 跳转到预留的 `/stores/add` 轻占位页，为后续接入真实新增流程预留承载点
- `平台配置`、`同步策略`、`接入说明` 在 V1 中先使用消息反馈或轻占位交互，不扩展为真实业务页面
- `立即同步` 和 `重新授权` 在 V1 中可以先表现为消息反馈或模拟状态变化

不在首页引入复杂的批量选择工作流；批量能力只保留最轻的 `批量同步`。

## Visual Language

首页应与已实现模块保持一致，而不是贴近原型的旧页面风格：

- 使用统一的工作台留白和 16px 左右的模块间距
- 保持克制、信息优先的中后台风格
- 信息密度高但层级要清晰
- 使用少量状态色强化异常，不让页面变成一片高饱和警报面

页面视觉重点顺序应为：

1. 标题与说明
2. 指标条
3. 待处理事项
4. 列表

## Testing

为 V1 补充以下测试：

- `src/data/storeManagement.ts` 的摘要统计、异常汇总、筛选和排序纯函数测试
- `/stores/list` 已从 placeholder route 切换为真实页面的路由级测试
- 首页关键文案、指标、待处理事项和表格列存在性的 source 测试

不在本轮引入端到端自动化测试。

## Rollout Notes

这是 `店铺管理` 的首页定义，不是整个店铺模块的全部设计。

后续推荐按以下顺序继续扩展：

1. `新增店铺`
2. `店铺详情 / 配置`
3. `店铺库存`
4. `利润核算`

这样可以保证首页先成为稳定入口，再逐步补全后续任务链路。
